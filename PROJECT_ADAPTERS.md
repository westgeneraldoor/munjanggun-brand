# PROJECT_ADAPTERS - 문장군 프로젝트별 적용 방식

> 버전: v1.4
> 최종 업데이트: 2026-07-01
> 목적: 여러 문장군 프로젝트가 중앙 브랜드 원본과 상품/자산 위키를 공유하되, 채널별 규칙과 적용 책임을 안전하게 분리하는 기준

## 0. 적용 책임 원칙

중앙 브랜드 총괄이 각 프로젝트 파일을 직접 고쳐 넣는 방식은 기본값이 아니다.

문장군 브랜드 적용은 아래 책임 분리로 진행한다.

| 역할 | 책임 |
| --- | --- |
| 중앙 브랜드 총괄 | 중앙 원본 유지, 적용 기준 제공, 하달 프롬프트 작성, 충돌 판단, 최종 검수 |
| 프로젝트 총괄 | 자기 프로젝트 구조 파악, 중앙 원본 연결 위치 결정, 프로젝트 내부 수정, 변경 파일 보고 |
| 검수 담당 | 금지 표현, 현장 판단, 사진/디자인 기준, 민감 정보, 토큰 사용 여부 확인 |

중앙 총괄은 긴급 보안 수정, 명백한 오탈자, 프로젝트 총괄의 명시 요청이 있는 경우를 제외하고 각 프로젝트 내부 파일을 직접 덮어쓰지 않는다.

프로젝트 총괄은 중앙 원본을 그대로 복사하는 것이 아니라, 자기 프로젝트의 구조와 자동화 흐름에 맞게 얇은 참조 문서나 어댑터를 만든다.

## 1. 기본 구조

문장군 프로젝트는 앞으로 아래 구조를 따른다.

```text
중앙 브랜드 원본
  BRAND_CONTEXT.md
  FIELD_JUDGMENT_RULES.md
  DESIGN.md
  DESIGN_QUICKSTART.md
  PHOTO_TREATMENT.md
  ANTI_PATTERNS.md
  tokens/brand.tokens.json
  tokens/brand.css
  EVIDENCE_REGISTER.md
  OPEN_QUESTIONS_REGISTER.md
  BRAND_WIKI_ARCHITECTURE.md
  SOURCE_REGISTRY.md
  PRODUCT_WIKI_INDEX.md
  ASSET_SEMANTIC_INDEX.md
  상품별 asset-manifest.json
  BRAND_MATERIAL_INDEX.md
  RAW_MATERIAL_INTAKE_PROTOCOL.md
  CUSTOMER_SEGMENTS.md
  PRODUCT_SELECTION_GUIDE.md
  FIELD_STORY_BANK.md
  REVIEW_PROOF_BANK.md
  PROOF_ASSET_INDEX.md
  FAQ_OBJECTION_BANK.md
  COPY_ASSET_BANK.md
  PROJECT_ADAPTERS.md

각 프로젝트
  중앙 원본 참조 문서
  프로젝트별 어댑터
  프로젝트별 운영/QA 문서
```

중앙 원본은 문장군이 무엇을 말해야 하는지, 어떤 현장 기준을 가져야 하는지, 어떻게 보여야 하는지 정한다.

프로젝트별 어댑터는 블로그, 인스타그램, 리뷰릴스, 시공릴스, 웹앱처럼 채널마다 다른 출력 형식과 운영 규칙을 정한다.

브랜드 원료 은행은 중앙 기준 아래의 공통 재사용 재료다. 고객 상황, 제품 선택 기준, 비식별 현장 이야기, 리뷰/증거 유형, FAQ, 카피 원료는 중앙에서 관리하되, 블로그 SEO, 인스타그램 장수와 해시태그, 릴스 컷 구성, 웹앱 구현 방식은 프로젝트 어댑터에 둔다.

## 2. 중앙 원본에 넣는 것

아래는 중앙 브랜드 폴더에 둔다.

- 회사 기본 정보
- 제품/서비스 범위
- 운영 체계
- 무료 방문 실측 기준
- 시공 가능 여부 판단 기준
- 현장 변수
- 고객 오해와 상담 언어
- 신뢰 근거
- 디자인 기준
- 금지 표현과 위험한 단정
- 리뷰 수, 가격, A/S, 일정 등 변동 claim의 근거표
- 고객 상황별 불안과 질문
- 제품별 선택 기준과 현장 확인 질문
- 비식별 현장 이야기 패턴
- 리뷰와 증거 자산의 안전한 사용 기준
- 상담 FAQ와 공통 카피 원료
- 연계 프로젝트 자료를 중앙 원료로 승격하는 절차
- 상품 상세페이지, 이미지/GIF, 썸네일 자산의 출처, 의미 태그, 사용 가능 상태, claim/privacy 위험

## 3. 프로젝트 어댑터에 넣는 것

아래는 각 프로젝트에 둔다.

- 채널별 글/영상/카드 구조
- 파일 저장 규칙
- QA 실행 방식
- 발행 승인 방식
- 통계 분석 방식
- 프로젝트별 제외 소재
- 채널별 제목/후킹/해시태그 규칙
- 채널별 이미지 삽입 방식
- 블로그 SEO 제목 공식의 세부 점수화
- 인스타그램 카드 장수, 해시태그, 업로드 규칙
- 릴스 컷 구성, 자막 타이밍, TTS, 렌더 설정
- 웹앱 라우팅, 컴포넌트 구현, 배포 방식

예를 들어 블로그 프로젝트의 `단일 발행 MD`, `네이버 태그란`, `일일 SEO 관제`, `AppSheet 현장 서사 슬롯`, `발행 하드게이트`는 중앙 원본이 아니라 블로그 어댑터에 둔다.

## 4. 충돌 처리

중앙 원본과 프로젝트 문서가 다르면 바로 덮어쓰지 않는다.

먼저 아래 네 가지로 분류한다.

| 분류 | 뜻 | 처리 |
| --- | --- | --- |
| 중앙 우선 | 브랜드 전체에 적용되는 기준 | 프로젝트 문서를 중앙 기준에 맞춘다 |
| 프로젝트 우선 | 채널 특성상 필요한 예외 | 프로젝트 어댑터에 명시한다 |
| 중앙 업데이트 필요 | 프로젝트에서 더 최신 정보가 발견됨 | 중앙 원본과 CHANGELOG를 업데이트한다 |
| 확인 필요 | 사실 여부나 운영 범위가 불확실함 | 사장 확인 전 발행/적용하지 않는다 |

## 5. 민감 정보 처리

중앙 브랜드 폴더에는 고객명, 전화번호, 상세 주소, 상담 원문, AppSheet 원본 레코드, 관리자 통계 원본을 넣지 않는다.

가능한 것은 비식별 요약이다.

예:

- "수원 구축 아파트 현장에서 좁은 현관 폭 때문에 3연동 가능 여부를 상담한 사례"
- "화장실문 세트 교체 시 타일공사 계획이 없어 PVC마감재를 안내하는 경우"

불가능한 것:

- 고객 실명
- 전화번호
- 동호수
- 상담 원문
- AppSheet 원본 캡처
- 네이버 관리자 통계 원본

## 6. 프로젝트 시작 시 필수 확인

새 문장군 프로젝트를 만들 때는 아래 순서로 확인한다.

1. 중앙 `BRAND_CONTEXT.md` 읽기
2. 중앙 `FIELD_JUDGMENT_RULES.md` 읽기
3. 중앙 `DESIGN.md` 읽기
4. 디자인/사진/구현 작업이면 `DESIGN_QUICKSTART.md`, `PHOTO_TREATMENT.md`, `ANTI_PATTERNS.md`, `tokens/` 읽기
5. 변동 claim이 있으면 `EVIDENCE_REGISTER.md` 읽기
6. 확인되지 않은 운영 기준이 있으면 `OPEN_QUESTIONS_REGISTER.md` 읽기
7. 상품, 상세페이지, 이미지, GIF, 썸네일 자산을 쓰면 `BRAND_WIKI_ARCHITECTURE.md`, `SOURCE_REGISTRY.md`, `PRODUCT_WIKI_INDEX.md`, 필요한 상품 위키, `ASSET_SEMANTIC_INDEX.md`, 상품별 `asset-manifest.json` 읽기
8. 고객/제품/현장/리뷰/FAQ/카피 원료를 쓰면 `BRAND_MATERIAL_INDEX.md`와 필요한 개별 원료 은행 읽기
9. 프로젝트 자료를 중앙으로 올릴 후보가 있으면 `RAW_MATERIAL_INTAKE_PROTOCOL.md` 읽기
10. 해당 프로젝트의 어댑터 문서 만들기
11. 기존 프로젝트 문서가 있다면 중앙 원본과 충돌 감사
12. 충돌을 `중앙 우선 / 프로젝트 우선 / 중앙 업데이트 필요 / 확인 필요`로 분류
13. 프로젝트 적용 결과는 프로젝트 내부 문서에 기록한다. 중앙 `CHANGELOG.md`에는 중앙 원본 변경 또는 중앙 승격이 확정된 내용만 기록한다.

## 7. 프로젝트 총괄 적용 보고 형식

각 프로젝트 총괄은 중앙 원본 적용 후 아래 형식으로 보고한다.

```text
프로젝트:
적용 일자:
읽은 중앙 문서:
수정한 파일:
새로 만든 참조/어댑터 문서:
중앙 원본과 충돌한 지점:
충돌 분류: 중앙 우선 / 프로젝트 우선 / 중앙 업데이트 필요 / 확인 필요
민감 정보 노출 여부:
사진/디자인 기준 적용 여부:
토큰 사용 여부:
상품/자산 위키 확인 여부:
자산 usageStatus/privacyStatus/claimRisk 확인 여부:
사용한 source_id / asset_id / claim_id / proof_id:
review_due 또는 재확인 필요 시점:
retired/superseded 자료 사용 여부:
외부 발행 가능 여부:
원료 은행 사용 여부:
원료 상태값 확인 여부:
변동 claim 근거 확인 여부:
확인 필요 항목 등록 여부:
남은 확인 사항:
```

중앙 총괄은 이 보고를 기준으로 중앙 원본 수정 필요 여부와 추가 검수 필요 여부를 판단한다.

## 8. 중앙 총괄이 직접 수정해도 되는 경우

이 절이 중앙 총괄 직접 수정 예외의 기준이다.

아래 경우에는 중앙 총괄이 프로젝트 내부를 직접 수정할 수 있다.

- 고객 개인정보, 토큰, `.env`, AppSheet 원본 등 민감 정보가 노출된 경우
- 중앙 원본 경로가 명백히 잘못되어 프로젝트가 깨지는 경우
- 프로젝트 총괄이 명시적으로 중앙 총괄에게 수정을 요청한 경우
- 단순 링크, 오탈자, README 참조 경로처럼 운영 위험이 낮고 범위가 좁은 경우

그 외에는 프로젝트 총괄이 직접 적용하고 중앙 총괄은 검수한다.
